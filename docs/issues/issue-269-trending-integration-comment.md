## Specific improvements to the plan (aligned with our codebase)

After reviewing the codebase, here are concrete adjustments and additions to the plan.

### 1. Terminology and scope

- **"Sample content ideas"** in our system = the `contentIdeas` array on each **audience scenario** generated by the website analysis pipeline (`services/openai.js` → `generateAudienceScenarios`, persisted in `audiences`).
- Each idea has `{ title, searchIntent, businessAlignment }`.
- Our pipeline cache is **30 days** at the org level (`CACHE_TTL_DAYS = 30` in `website-analysis-pipeline.js`), not 1 week. The issue’s “1-week” should be updated to match, or we introduce a separate TTL for trending vs. static ideas.

### 2. X (Twitter) API vs. Grok

- We already use **xAI Grok** (`services/grok-tweet-search.js`) for tweet search via the Agent Tools API (`XAI_API_KEY`).
- Grok searches for **topic-specific tweets** (e.g. for a blog post). It is **not** a “trending hashtags” API.
- **Recommendation:** Phase 1 should either:
  - **Option A:** Extend Grok with a new prompt (e.g. “Return trending hashtags/topics in niche X”) if that fits the model’s tools.
  - **Option B:** Add Twitter API v2 for trending hashtags only if Grok cannot provide them. Avoid duplication for topic-specific tweet search.

### 3. Reuse existing integrations

| Source      | Status                                      | Where used                               |
|-------------|---------------------------------------------|------------------------------------------|
| News API    | Already integrated                          | `services/news-article-search.js`        |
| Grok (xAI)  | Already integrated                          | `services/grok-tweet-search.js`          |
| YouTube     | Already integrated                          | `services/youtube-video-search.js`       |

- **News API:** Can be reused for breaking news and “trending” news by topic. Add a thin wrapper for “trending for niche” that calls `searchRelevantArticles` with trending-oriented params (e.g. sort by recency/popularity).
- **Grok:** Already handles graceful degradation when `XAI_API_KEY` is unset. New trending use cases should follow the same pattern.
- **Google Trends:** We do not have this yet. Consider serpapi.com or `google-trends-api` (unofficial). Document cost and rate limits before Phase 2.

### 4. Caching

- **Current use of Redis:** BullMQ (job queue) and pub/sub for streams (`stream-manager.js`, `jobs:{jobId}:events`). No general-purpose key-value cache for external API responses.
- **Recommendation:** Introduce a small Redis cache layer (or use our existing connection via `job-queue.js`’s `getConnection`) with keys like `trending:hashtags:{nicheHash}` and `trending:news:{nicheHash}`, 1-hour TTL as in the issue.
- **Performance target:** “Trending data fetch &lt; 1s” — realistic only if we serve from cache. First request may be slower (e.g. Grok 10–30s). Consider:
  - A background cron to warm cache for common niches.
  - Or a “best effort” approach: return cached trending if available; otherwise use static ideas without blocking.

### 5. Where to inject trending

- **Option A (enrich prompt):** Add trending data (hashtags, news headlines) to the `generateAudienceScenarios` prompt so `contentIdeas` are generated with trending context.
- **Option B (separate endpoint):** Add `GET /api/v1/strategies/:id/sample-ideas?refresh=true` that:
  - Fetches cached or fresh trending data.
  - Merges it with existing audience `contentIdeas` or regenerates ideas via a lightweight prompt.
- **Option C (hybrid):** Use Option A for new analyses, Option B for “Refresh sample ideas” after signup.

Recommendation: Start with Option B for “Refresh sample ideas” (no change to full pipeline), then add Option A when we’re confident in trending data quality.

### 6. Data flow for “Refresh sample ideas”

- The “Refresh” action should:
  1. Accept `audienceId` (or strategyId) and optionally `organizationId`.
  2. Fetch trending data (hashtags, news) scoped to that audience’s niche.
  3. Call a new OpenAI helper (e.g. `regenerateContentIdeasWithTrends(audience, trendingData)`) that returns an updated `contentIdeas` array.
  4. Persist to `audiences.content_ideas` or a new `audience_content_ideas` table if we want history.
  5. Return updated ideas to the frontend.

- Currently `audiences` stores scenarios as JSONB (`customer_scenarios` in `organization_intelligence` or per-audience rows). We should confirm the exact schema for `contentIdeas` before adding a refresh flow.

### 7. Env vars

- Add to `.env.example` and docs:
  - `XAI_API_KEY` (Grok) — used today but not documented in `.env.example`
  - `NEWS_API_KEY` — already in `.env.example`
  - `GOOGLE_TRENDS_API_KEY` or `SERPAPI_KEY` — if we add Google Trends in Phase 2

### 8. Cost and reliability

- **Grok:** Agent Tools API can be slow (10–30s). Avoid using it for per-request trending unless we accept latency or use a dedicated cache.
- **News API:** Free tier is limited (e.g. localhost only, 100 requests/day). Document limits and consider a paid plan for production.
- **Fallback:** If any trending API fails, we should fall back to existing static `contentIdeas` without failing the whole request. Our existing services already do this (e.g. Grok returns `[]` when no key).

### 9. Implementation order

- **Phase 1 (MVP):**
  1. Redis cache module for trending data (key pattern, TTL).
  2. Trending fetch service that uses News API (and optionally Grok for hashtags if feasible).
  3. `POST /api/v1/audiences/:id/refresh-sample-ideas` (or equivalent) that fetches trending, calls `regenerateContentIdeasWithTrends`, and returns new ideas.
  4. Basic trend score/badge: if we have trending data for an idea’s topic, mark it as “trending” (binary for MVP).

- **Phase 2:** Google Trends, richer trend scoring (0–100), velocity.
- **Phase 3:** Auto-regeneration, personalization, A/B testing.
